Agent 시대의 업무 방식

나: "로그 파싱 모듈 리팩토링해줘. 기존 테스트 유지하고, 새 기능 테스트도 추가해." AI: 파일 분석 중... → 기존 테스트 확인 → 리팩토링 진행 → 새 테스트 작성 → 전체 테스트 실행 (32/32 pass) → 결과 보고 나: (diff 검토 후 승인)

Pilot

가장 중요한 요소 = 당신

Pilot의 핵심 역량

  • 문제 분해 — 복잡한 문제를 AI가 처리 가능한 단위로
  • 실패 패턴 인식 — 할루시네이션, 컨텍스트 한계 예측
  • 도메인 지식 — AI 출력의 적절성을 즉시 판단
  • 파이프라인 설계 — 반복 가능한 워크플로우 구축

Single Agent

1명이 모든 관점

Multi-Agent

Agent 유형

Main Agent (주 에이전트) - 사용자와 직접 대화 - 전체 워크플로우 조율 - Sub-Agent 호출 및 결과 종합 Sub-Agent (서브 에이전트) - 특정 작업에 특화 - Main Agent에 의해 호출됨 - 독립적으로 작업 후 결과 반환 예시: Feature-Dev의 3개 서브에이전트 - code-explorer: 코드베이스 탐색 전문 - code-architect: 아키텍처 설계 전문 - code-reviewer: 코드 리뷰 전문

# 병렬로 3개 Agent 실행 다음 에이전트들을 병렬로 실행: Agent 1: 완전성 검증자 → 요구사항 누락 여부 확인 Agent 2: 명확성 검증자 → 모호한 표현 확인 Agent 3: 실현가능성 검증자 → 기술적 구현 가능성 확인 모든 에이전트 완료 후 결과 종합

Hookify 동작 메커니즘

1. 규칙 파일 로드 ~/.claude/hookify/rules/*.md 자동 스캔 각 .md 파일이 하나의 prevention rule 2. Hook 트리거 PreToolUse → 도구 실행 전 규칙 검사 PostToolUse → 도구 실행 후 결과 검증 Stop → 세션 종료 시 최종 검사 UserPromptSubmit → 사용자 입력 검증 3. 규칙 위반 시 BLOCK → 동작 차단 + 이유 설명 또는 conversation-analyzer 서브에이전트 호출

Hookify 규칙 파일 예시

# ~/.claude/hookify/rules/no-secrets.md --- name: no-secrets-commit hook: PreToolUse matcher: Bash --- # 시크릿 커밋 방지 ## 규칙 git commit 명령 실행 전 다음을 확인: - .env 파일이 포함되어 있지 않은가? - API 키가 하드코딩되어 있지 않은가? - credentials.json이 스테이징되어 있지 않은가? ## 위반 시 BLOCK: 시크릿이 포함된 것 같습니다. 확인 후 제외하세요.

Hookify 서브에이전트: conversation-analyzer

역할: - 규칙 위반 여부를 심층 분석 - 단순 패턴 매칭이 아닌 맥락 기반 판단 호출 시점: - PreToolUse에서 규칙 위반 의심 시 - 복잡한 판단이 필요한 경우 응답 형식: { "decision": "ALLOW" | "BLOCK", "reason": "판단 근거 설명", "confidence": 0-100 }

from claude_code_sdk import Agent, Swarm # 여러 에이전트 정의 reviewers = [ Agent(name="security", prompt="보안 관점 리뷰"), Agent(name="performance", prompt="성능 관점 리뷰"), Agent(name="quality", prompt="코드 품질 리뷰") ] # Swarm으로 병렬 실행 swarm = Swarm(agents=reviewers) results = await swarm.run_parallel( task="src/gc_module.c 리뷰" ) # 결과 집계 for r in results: print(f"{r.agent_name}: {r.output}")

Command vs Skill 비교

Command Skill
뭐냐 단축키 지식
누가 내가 실행 Claude가 참조
언제 /명령어 입력 시 작업 전 자동
왜 긴 프롬프트 줄이려고 Claude가 맥락 알게

Command만 있으면?

# Command만 있는 경우 나: /review auth.py Claude: (리뷰는 하는데 우리 팀 규칙을 모름) "이 함수명 camelCase 추천합니다" ← 우리 팀은 snake_case 쓰는데...

단축키는 있지만 맥락(규칙)이 없어서 엉뚱한 결과

둘 다 있으면?

# Command + Skill 조합 나: /review auth.py Claude: (Skill 읽고 우리 규칙 파악) "snake_case 규칙 위반입니다" ← 정확!

단축키(Command)로 호출 + 규칙(Skill)으로 맥락 = 완벽한 조합

체계적인 개발 파이프라인

10-Phase 플로우

[1.INIT] 상태 파일 + 훅 등록 [2.ANALYZE] 요구사항 분석 (REQ/AC 정의) [3.REQ-VERIFY] ★ 병렬 3개 에이전트 (완전성/명확성/실현가능성) [4.EXPLORE] 코드베이스 탐색 (병렬 3개) [5.DESIGN] 아키텍처 설계 (병렬 3개 접근법) [6.DESIGN-VERIFY] ★ 병렬 3개 에이전트 (타당성/호환성/유지보수성) [7.IMPLEMENT] 코드 작성 [8.CODE-VERIFY] ★ 병렬 4개 에이전트 (verify-work) [9.TEST] 테스트 + 커버리지 90%+ [10.COMPLETE] 완료 보고서 + 훅 제거

신뢰도 기반 필터링

# 신뢰도 임계값 설정 THRESHOLDS = { SECURITY_VULNERABILITY: 90% MISSING_REQUIREMENT: 85% TEST_COVERAGE_GAP: 80% CODE_QUALITY: 75% } # 필터링 규칙 신뢰도 < 임계값 → FILTERED OUT (노이즈 제거) 신뢰도 >= 임계값 → REPORTED (보고)

False Positive 제거로 개발자 집중도 향상
